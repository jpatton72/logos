Pre-rename cleanup pass and a salvage migration path for users whose data still lives under the old `logos` directory.

## Salvage migration for legacy `logos` data dirs

The original logos→aletheia migration only ran on the very first launch of the new app. If you launched 0.1.7 or 0.1.8 once on Linux, the (broken) seed created an empty `~/.local/share/aletheia/` and from that point on the migration silently skipped — your real data, including notes and bookmarks, stayed stranded under `~/.local/share/logos/`.

0.1.10 detects this situation on launch:

- New Aletheia data dir exists but is **pristine** — no books, no notes, no bookmarks.
- Legacy `logos` data dir exists with **real data** — at least one row in its `books` table.

When both are true, the empty Aletheia dir is removed and the legacy dir is migrated in normally (`logos.db` → `aletheia.db`, all notes/bookmarks/preferences carried over). Real installs are never touched: any non-zero count on books/notes/bookmarks aborts the salvage path.

## Code cleanup: pre-rename leftovers

Removed or updated the rename leftovers that had nothing to do with the migration shims:

- `LOGOS_AI_MAX_TOKENS`, `LOGOS_AI_RATE_LIMIT`, `LOGOS_AI_RATE_WINDOW_SECS` env vars renamed to `ALETHEIA_AI_*`. The old names still work as a fallback so existing shell configs aren't broken.
- Ingest script `User-Agent` strings switched from `LogosBibleApp/1.0` to `Aletheia/1.0` so upstream maintainers see who's actually fetching their data.
- `generate_icon.py` and `scripts/_user_check.py` print/error messages updated to say *Aletheia*.

What was deliberately **not** renamed:

- Disclaimers about Faithlife's *Logos Bible Software* product line — required for trademark distinction.
- SBLGNT GitHub URLs at `github.com/LogosBible/SBLGNT` — that's the actual upstream org for the SBL Greek New Testament.
- The Greek word λόγος / "logos" in mock data and SPEC examples — genuine biblical content (Strong's G3056).
- The legacy migration code itself (`migrate_legacy_data_dir`, `migrate_legacy_keyring_entries`, the `'logos-app-state'` localStorage key) — load-bearing for the user-data move.

## Upgrade

Standard upgrade path. Once 0.1.10 launches, the salvage migration will pick up any stranded `~/.local/share/logos/` data on Linux (or `%APPDATA%\logos\` / `~/Library/Application Support/logos/` if anyone ever ran the old build elsewhere). No manual cleanup required.
