Fix for the v0.1.3–v0.1.5 hover-highlight bug: tables added by later schema migrations weren't getting populated for users upgrading from earlier versions.

## Install (Windows)

Download `Aletheia_x64-setup.exe` (always points at the latest stable build) or the version-pinned `Aletheia_0.1.6_x64-setup.exe` below. Upgrades v0.1.0–v0.1.5 in place. **First launch after this upgrade will run a one-time backfill** that copies the new data into your existing database — it's silent and takes a second.

## What was broken

The bundled-database seeding code only fires when you install Aletheia for the first time and have no existing user data. The schema migrations correctly *create* new tables on upgrade, but had no way to *populate* them — they stayed empty. So `english_word_alignment` (added in v0.1.3 for hover-to-highlight) and `english_strongs_index` (added in v0.1.2 for the Lexicon's English-lookup tab) were both empty for anyone upgrading from v0.1.1 or earlier.

Visible symptoms:
- Hover-to-highlight in the reader did nothing (no English tokens to highlight).
- Lexicon's English Word tab returned no results for any query.

## What changed

- **Startup backfill from the bundled DB.** On launch, if `english_word_alignment` or `english_strongs_index` are empty in the user's database but populated in the bundled database that ships with the installer, copy the missing rows over. The copy uses content-based joins on book + chapter + verse + translation, so verse-ID drift between bundled and user databases doesn't matter.
- **Idempotent.** Each table is checked individually; once populated, the backfill is a no-op on subsequent launches.
- **Best-effort.** Any failure logs and is swallowed — a corrupt or missing bundled DB can't block startup.
- Removed the temporary `[hover-highlight]` console diagnostic added in v0.1.5.
