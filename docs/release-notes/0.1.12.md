Second attempt at the distorted/maxed-volume verse narration fix. The 0.1.11 fix wasn't enough.

## What 0.1.11 missed

0.1.11 assumed Piper's stdout output was always raw PCM and unconditionally wrapped it in a constructed WAV header, with a passthrough escape hatch when the output already started with `RIFF`. Testing showed the audio was still distorted and maxed-volume. The likely failure mode: Piper *does* emit a RIFF header on stdout but with bogus length fields (it can't seek back to write the final data size on a non-seekable stream), or with `audio_format = 3` (IEEE float) instead of `1` (PCM). The browser then either rejects the file or interprets PCM samples as float values, which clip and produce the loud-distorted output we kept hearing.

## 0.1.12 fix

Stop fighting Piper's stdout format. Piper now writes a real WAV file to a temp path on disk; we read the file back, return its bytes, and clean up. Piper handles the WAV format itself with all header fields correct (it can seek back to write the final size, can pick PCM as the format, can match the model's sample rate). The browser gets a properly framed WAV that decodes correctly without any header surgery on our side.

Concurrent synth requests get unique temp filenames (pid + atomic counter) so two rapid Listen clicks across different verses can't collide. Stale files from a crash are harmless — the OS cleans `/tmp` itself.

This drops the WAV-header construction code from 0.1.11 entirely; that path was wrong from the start and the dead code wasn't worth keeping.

## Upgrade

Standard. No re-download of Piper or the voice model needed — the fix is purely in how we invoke Piper.
